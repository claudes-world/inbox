# Inbox SQL Schema & Migrations

This directory is the single source of truth for the Inbox database schema.

## Files

- `001-init.sql` — initial schema (addresses, messages, conversations, deliveries, events, etc.). Do not edit this file after release; add forward migrations instead.
- `002-example-placeholder.sql` — no-op sample migration demonstrating the runner.
- `seed.sql` — development seed data. Not a schema change, not tracked by the migration runner.

## How migrations work

The BFF (`packages/bff/src/migrations.ts`) runs a simple forward-only migration system on startup:

1. Ensures a `schema_migrations` tracking table exists.
2. Lists every file in `schema/` matching `NNN-*.sql` (three-digit prefix).
3. Applies each file that isn't already recorded in `schema_migrations`, in lexical order, inside a transaction.
4. Records the version on success.

### Tracking table

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  version TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at INTEGER NOT NULL
) STRICT;
```

### Adding a new migration

1. Pick the next version number (e.g. `003`).
2. Create a file named `003-short-description.sql`. Use kebab-case after the prefix.
3. Write plain SQL. Keep each migration small and self-contained.
4. Restart the BFF (`systemctl --user restart inbox-bff.service` or re-run the dev process). The runner will pick up and apply the new file.

### Rules of thumb

- **Forward-only, no rollbacks in v1.** If a migration is wrong, ship a new migration that fixes it.
- **Never edit an already-released migration file.** Other databases may have already applied it.
- **One concern per file** — easier to reason about and easier to recover from.
- Migrations run in a transaction, so a failing statement rolls the whole file back. The version is not recorded on failure, so the runner retries next startup.

### Backward compatibility

If a pre-existing database already has the `addresses` table but no `schema_migrations` entries, the runner records `001` as applied before doing anything else. This keeps existing dev/prod databases safe without re-running `001-init.sql`.
