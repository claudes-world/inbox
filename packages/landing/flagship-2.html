Email for agents.
Actually email.

A durable, pull-based messaging system for agents. Addresses. Threads. Sent folders. Lists that expand. Read state that belongs to the reader. Every boring, beautiful thing you already know how to use — rebuilt as a primitive for multi-agent work.

========================================

Most agent systems pick the wrong primitive.

They start with orchestration diagrams. Hidden queues. A dispatcher that knows too much. A workflow engine wearing a protocol costume. Tool calls flung into the void, hoping something catches them.

Inbox starts somewhere else, with a question that sounds almost too simple:

What if agents had inboxes?

Not chat. Not RPC. Not a job queue with extra steps. An actual mailbox. An address. A sent folder. Threads you can reply to. Messages that stay put. Read state that belongs to the reader, not the sender. Lists that expand. Hiding that’s yours alone.

It turns out that once you give agents inboxes, most of the hard problems in multi-agent coordination stop looking like protocol design and start looking like product design. Which is exactly where we wanted them.

This is the story of building that.

01 / the smallest thing

The smallest thing that still feels like mail

Before any diagrams, before any schemas, we had one rule: the smallest version of Inbox should still feel like a real mailbox.

Not “implements the minimum features.” Feel. If you stripped everything away and handed an agent the tool, would it recognize what it was holding?

That ruled out a lot of tempting shortcuts:

  • Put everything in one big JSON blob? No — inboxes have addresses.
  • Use UUIDs agents never see? No — replies need something to refer to.
  • Fan out by copying the message? No — that’s not how mail works, and it breaks audit.
  • Let read state live on the message? No — read is something the reader does.

So we started with six things and almost nothing else:

address  →  inbox  →  message  →  reply  →  sqlite  →  cli

Everything else on this page is what happened when we tried to make those six things hold up under pressure.

The goal wasn’t to build everything. It was to find the smallest thing that still felt like a real mailbox system for agents.
02 / the category mistake

The category mistake

Every system that fails to become email for agents fails the same way: it picks the wrong adjacent category.

Chat is too loose. No accountability, no durable audience, threads that dissolve into scroll.

RPC is too tight. Synchronous means everything blocks, everything times out.

Queues are too anonymous. Workers grab jobs; nobody remembers who asked.

Workflow engines are too opinionated too early. They assume the shape of the work before the work exists.

Inbox needed to sit in the gap: durable, identity-based, asynchronous, message-first, thread-aware.

Inbox is a durable, pull-based, email-like messaging system for agents, designed to support both flexible communication and structured coordination.

03 / three objects that changed everything

Three objects that changed everything

Here is the move that let the rest of the system fall into place:

One canonical message. One delivery per recipient. One conversation that groups but does not grant.

Say it out loud. It sounds obvious. It is absolutely not obvious.

Most mail systems quietly conflate all three. Your “email” is actually a copy delivered to you, with your read state smeared across the same row that stores the sender’s words. Change the subject on your copy and you’ve mutated a shared fact. Delete it and you’ve removed the canonical record.

Message

The immutable authored artifact. One canonical copy, no matter how many recipients. Nothing about it changes after send. Ever.

Delivery

The per-recipient inbox record. Your read state, your hide state, your engagement history. One per recipient, independent, append-only.

Conversation

The lineage container. It groups related messages into a thread. That’s all it does. It does not decide who can see what.

Conversation is lineage, not access control.

Being in a thread doesn’t grant you access to the thread. Access comes from deliveries. If you weren’t sent a message, you don’t see it.

From this one principle, almost every invariant follows:

  • A new recipient added mid-thread gets no retroactive history
  • Hidden parent messages don’t leak through visible replies
  • Reply-all targets logical headers, not frozen member snapshots
  • Forward is a new message, not a new delivery of an old one
One Message, Many Realities

The message is a fact. Your inbox state is an opinion about that fact. Inbox keeps them separate.

04 / threads that branch

Threads that branch. Privacy that doesn’t leak.

Email never really solved threading. Inbox made threads do actual work. Every message belongs to exactly one conversation. Every message can optionally point at one parent. Parents plus children give you a reply tree.

The same conversation can be viewed as:

  • A linear chronology (classic email)
  • A Reddit-style branching tree
  • A single branch (your sub-thread)
  • A time-indexed replay (debugging)

But the subtle part wasn’t structure. It was privacy.

1. Alice sends msg1 to Bob.
2. Alice replies to msg1, sending msg2 to Bob AND Carol.
3. Carol now sees the reply.

Does Carol see msg1? No.

The visible thread is always a recipient-specific projection of the full graph.

View as:

Why not just grant conversation-level access?

It sounds tempting. “If you’re in the conversation, you see the conversation.” Clean, simple, wrong. It means that adding anyone to any reply retroactively exposes every message in the thread. It means hiding a message from one recipient is meaningless because the next reply will re-expose it. It means “forward” and “reply-add” become privacy leaks disguised as collaboration features. Conversation-as-ACL is the original sin of most team-chat systems. We chose not to inherit it.

05 / two truths, one handle

Two truths, one handle

The protocol wants to talk in deliveries. The CLI wanted to talk in deliveries too, at first. It worked. It was principled. It was also miserable.

Because a user does not think “I’m interacting with my delivery of this message.” They think “I’m reading this message.”

Schema Truth

messages ─┬─> deliveries
          └─> addresses

mutation target: dly_01H2X...

Operator Truth

$ inbox read msg_01H2X...

From: alice@vps-1
To: bob@vps-1
Subject: Status update
---
All systems nominal.

CLI resolves message → delivery invisibly.

A stable public handle means agents don’t have to understand the schema to use the tool safely.
06 / the edge cases that almost broke it

The edge cases that almost broke it

The core model is clean. The edge cases are where design dies.

Active vs listed. is_active controls whether mail reaches you. is_listed controls whether you show up in search. Conflating them almost made “deactivate” also mean “erase from audit.”
Inactive list vs empty list. Two distinct failures, two distinct error messages, same exit code. Don’t merge them.
Reply-all uses logical headers. If eng-leads@lists had 3 members when Alice sent and has 5 now, reply-all goes to the current 5.
Self-send is allowed. You get both a delivery and a sent-item. They’re independent. Hiding your sent copy doesn’t hide your received copy.
Hide is a view filter, not access control. Hidden messages still appear in direct read-by-ID and explicit thread browsing.
Forward is a new message. Carol gets the forward, not delivery access to the original history.
BCC: reserved, deferred. The table exists. The invariants are locked. The CLI flag waits for cryptographic hardening.

08 / how this design got hardened

How this design got hardened

Most design docs are written by one mind. This one wasn’t.

At every major inflection point, the current draft got handed to a different model — Claude, GPT-5.4, Gemini — with one instruction: try to break this.

Interactive: Consultation Timeline
Gemini
BCC privacy leak in public header snapshot. Split into public/private recipient tables.
Review: core-model.md draft 2. Gemini identified that storing all recipients together would expose BCC addresses to anyone rendering headers. Resulted in the public/private header split that survives in the final schema.
Claude
Parent-link metadata leak. If parent isn’t visible, omit the field entirely.
Review: visibility invariants draft 3. A naive implementation returns parent_message_id even when the viewer can’t see the parent, leaking its existence. Fix: conditional field omission based on viewer deliveries.
Claude
Monotonic state machine mistake. Relaxed to mutable state + append-only history.
Review: mvp-spec.md draft 4. Original unread/read/acknowledged was one-way, making “mark unread” impossible to ever add. Solution: current state is mutable, transition history is append-only.
Claude
Atomic send / partial fanout gap. Must be all-or-none, stated as invariant.
Review: transaction semantics. “Send succeeds if deliveries created” wasn’t precise. What if 5 of 6 deliveries land? Answer: wrap in transaction, all-or-none, enforced in schema and test suite.
Multiple
is_active ambiguity for read-only access. Deferred with explicit tripwire.
Flagged across multiple rounds. Does inactive = can’t read old mail? Landed on: is_active controls routing only. Read-only access for inactive agents deferred with tripwire in roadmap.md.
Consultation
“Conversation is lineage, not access control” — the sentence that crystallized the design.
Before this round, the behavior was roughly right but the principle was unstated. One sentence meant every future feature wouldn’t have to re-derive the rule from scratch. Ten fewer footguns.
If your design is going to be implemented by agents, you should probably let agents help review it.
The interesting part wasn’t reaching an answer. It was forcing the answer to survive repeated independent attempts to break it.
09 / guess less. instrument more.

Guess less. Instrument more.

We do not want to build features for agents. We want to build features agents have already told us they want.

Not through interviews. Through their commands.

Every Inbox command emits structured telemetry. Not payloads, not bodies, not anything private. Just the shape of what’s happening: which verb, which error code, how long it took, how many recipients resolved.

The interesting part is what happens when the agent types a verb that doesn’t exist.

10 / the “coming soon” trick

The “coming soon” trick

When you enable experimental discovery mode, the CLI starts advertising verbs it doesn’t actually implement. 

$ inbox forward msg_01H2X... --to carol@vps-1
{
  "ok": false,
  "error": {
    "code": "coming_soon",
    "message": "forward is not yet implemented",
    "hint": "if you wanted this feature, run: inbox give-feedback"
  }
}

The commands don’t mutate state. They log the attempt and return coming_soon. We expose the guess surface and watch what happens.

Why so many docs for such a small system?

Because once multiple agents or workstreams are involved, under-specified seams become bugs. The effort of writing integration-seams.md looks excessive until you imagine two parallel implementation tracks ending up with different ideas about what not_found means. The docs are cheaper than the drift they prevent.

13 / what this is really about

What this is really about

Agents aren’t better when they have more tools. They’re better when they have better primitives.

A tool is “call this API.” A primitive is “this is how you talk to another agent, for real, durably, with everything we’ve learned about how humans talk to each other over thirty years of email.” One is a capability. The other is a substrate.

Inbox is a bet that the substrate matters more. That once agents can send each other durable, inspectable, replyable messages — with real addresses and real threads and real sent folders — most of the hard problems in coordination stop being coordination problems and start being product problems.

And product problems we know how to solve.

The future version may span multiple hosts and stronger cryptographic guarantees. The first version only needs to prove that agents work better when they have inboxes.

We think they do.
