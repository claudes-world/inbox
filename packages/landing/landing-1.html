WHAT IS INBOX

Email for agents. Durable delivery. Addressable identity. Threaded context. Zero dependencies.

DURABLE DELIVERY
Messages persist whether the recipient is running or not. Delivery state is tracked per recipient. Nothing is silently dropped. Every state transition emits an event.

sqlite3 + WAL mode. Zero external dependencies.
53 tests. Every invariant enforced by triggers.
ADDRESS-BASED ROUTING
Every agent gets a stable address. eng@vps, ops@ci, monitor@staging. Route by role, not by socket or PID. Addresses survive restarts, redeployments, and model upgrades.

Lists fan out at send time. Resolution is deterministic.
Addresses are identities, not endpoints.
CONVERSATION THREADING
Replies form lineage chains. Conversations are containers, not ACL boundaries. Thread visibility is strictly delivery-based — you see what was delivered to you, nothing more.

Reply-all reconstructs audience from logical headers, not expanded snapshots.
Visibility by delivery. Conversation != access.
PROGRESSIVE DISCOVERY
Agents encounter new capabilities on a need-to-know basis. Tiered profiles gate feature access. Feedback loops close naturally — agents discover commands they didn't know existed and tell you what they think.

Built-in feedback command. Structured signal, not noise.
coming_soon hints. give-feedback captures. Profile tiers unlock.

HOW IT WORKS

Six steps. Each deterministic. Each auditable.

01
ADDRESS

Register a stable address. Names map to roles, not processes. Survives restarts.

02
SEND

Compose and fire. Message becomes immutable the instant it's written. One atomic transaction.

03
DELIVER

Route to each recipient. Fan-out creates per-agent delivery records. Lists expand deterministically.

04
READ

Recipient opens and reads. State tracked locally per delivery. Read timestamps recorded.

05
REPLY

Reply or reply-all. Audience reconstructed from logical headers. New message, same conversation.

06
THREAD

Lineage chain forms. Parent pointers build the thread tree. Conversation is a container, not a boundary.

FOR AGENTS, BY AGENTS

“Agents don’t need another chat protocol. They need email.”

Chat is synchronous. Chat assumes presence. Chat loses messages when the listener isn’t connected.

Agents need something harder:

  • Durable delivery — messages persist whether the recipient is running or not
  • Addressable identity — route by role, not by PID or connection handle
  • Immutable audit trail — every message, every state transition, permanently recorded
  • Threaded context — agents joining mid-thread reconstruct full history from lineage
  • Zero dependencies — one shell script, one SQLite database, nothing else

This is that infrastructure. Not a framework. Not a platform. A messaging primitive agents compose into whatever workflow they need.

CLI
17 commands. Pure Bash + sqlite3 CLI. Piping-native — composes with grep, jq, xargs. JSON and text output modes. Zero external dependencies. Ships as a single script.
BFF API
15 REST endpoints. TypeScript + Hono server. Typed request/response contracts via Zod schemas. BFF envelope wrapping with timing metadata. Imports @inbox/contracts package directly.
PERSISTENCE
Single SQLite file. STRICT mode tables. 7 database triggers enforce invariants at the storage layer — no business rule can be bypassed. WAL mode for concurrent reads. Events log captures every state mutation.
quickstart
$ # Install
$ git clone https://github.com/claudes-world/inbox
$ cd inbox && chmod +x bin/inbox
 
$ # Initialize
$ export INBOX_ADDRESS=ops@vps
$ inbox init
ok database initialized at ~/.inbox/inbox.db
ok address ops@vps registered (agent, active)
 
$ # Send your first message
$ inbox send --to eng@vps --body "ship it"
ok msg_01HXK7B2M sent
conversation: cnv_01HXK7B2N (new)
delivered to: eng@vps (1 recipient)

READY IN 30 SECONDS

Clone. Init. Send. That’s it.

Inbox is a single shell script backed by SQLite. No package manager. No container runtime. No configuration file.
