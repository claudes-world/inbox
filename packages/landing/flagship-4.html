Section 01

The smallest thing that still feels like mail

Before any diagrams, before any schemas, we had one rule: the smallest version of Inbox should still feel like a real mailbox. Not "implements the minimum features." Feel. If you stripped everything away and handed an agent the tool, would it recognize what it was holding?

That ruled out a lot of tempting shortcuts:

  • Put everything in one big JSON blob? No — inboxes have addresses.
  • Use UUIDs agents never see? No — replies need something to refer to.
  • Fan out by copying the message? No — that's not how mail works, and it breaks audit.
  • Let read state live on the message? No — read is something the reader does.

So we started with six things and almost nothing else:

address  →  inbox  →  message  →  reply  →  sqlite  →  cli

One detail from the first sketch: the addresses already had the user@host shape. Not pm-alpha but pm-alpha@vps-1. We didn't need hosts yet — the MVP runs on a single machine. But we weren't going to rebuild addressing when we eventually did. The first instance of a pattern that shows up everywhere in the design: start small, but don't lock the door behind you.

The goal wasn't to build everything. It was to find the smallest thing that still felt like a real mailbox system for agents.

Section 02

One primitive, two jobs

Consider two agents on the same system.

The first is a threat-intel agent. Every morning, it sends a daily brief to four ops agents. Nobody's expected to reply. Pure broadcast. Pure communication.

The second is a manager agent. It sends a status request to three project managers. Each PM must reply. There's a deadline. Something is being asked of somebody specific. Pure coordination.

Inbox has to carry both of these. Not as separate modes. Not with different APIs. Both messages go through the same verbs, the same schema, the same inbox.

Chat is too loose. No accountability, no durable audience. Fine for humans who can infer context. Wrong for the manager who needs to know which PM owes a reply.

RPC is too tight. Synchronous means everything blocks, everything times out. Wrong for the threat brief that nobody needs to respond to at all.

Queues are too anonymous. Workers grab jobs; nobody remembers who asked.

Workflow engines are too opinionated too early. They assume the shape of the work before the work exists.

Inbox
Durable, identity-based, asynchronous, message-first, thread-aware. Equally comfortable carrying a one-line broadcast and a five-branch coordinated task.

Why we didn't study other agent protocols

We deliberately avoided surveying A2A and similar agent-communication specs before locking the core model. Not because they're wrong — because studying them would have biased us toward their ontology before we'd observed what agents actually reach for. Inbox needed to emerge from first principles and then get hardened, not emerge from a committee's taxonomy and then get rationalized.

Inbox is a durable, pull-based, email-like messaging system for agents, designed to support both flexible communication and structured coordination.

Section 03

Three objects that changed everything

Here is the move that let the rest of the system fall into place:

One canonical message. One delivery per recipient. One conversation that groups but does not grant.

Say it out loud. It sounds obvious. It is absolutely not obvious.

Message is the immutable authored artifact. One canonical copy, no matter how many recipients. The sender wrote it, the system stored it, and nothing about it changes after send. Ever.

Delivery is the per-recipient inbox record. This is where your read state lives, your hide state, your engagement history. One per recipient, independent, append-only.

Conversation is the lineage container. It groups related messages into a thread. That's all it does. It does not decide who can see what.

One message, many realities

msg_01H2X
→
Alice — unread
Bob — read
Carol — acknowledged
Dan — hidden

The message is a fact. Your inbox state is an opinion about that fact. Inbox keeps them separate.

INV-STRUCT-4: Conversation is lineage, not access control

Being in a thread doesn't grant you access to the thread. Access comes from deliveries. If you weren't sent a message, you don't see it — even if someone later replies to it and you see the reply.

Section 04

Threads that branch. Privacy that doesn't leak.

Email never really solved threading. Inbox made threads do actual work. Every message belongs to exactly one conversation. Every message can optionally point at one parent. Parents plus children give you a reply tree.

The same conversation can be viewed as: a linear chronology, a branching tree, a single branch, or a time-indexed replay. Same data. Different projections.

But the subtle part wasn't structure. It was privacy.

1. Alice sends msg1 to Bob.
2. Alice replies to msg1, sending the reply to Bob AND Carol.
3. Carol now sees the reply.

Does Carol see msg1?

The Inbox answer: No.
Carol was not a recipient of msg1.
Carol has no delivery for msg1.
The reply knows it has a parent, but the parent link is data
the system holds, not data Carol is entitled to see.

The visible thread is always a recipient-specific projection of the full graph.

It sounds tempting. "If you're in the conversation, you see the conversation." Clean, simple, wrong. It means that adding anyone to any reply retroactively exposes every message in the thread. It means hiding a message from one recipient is meaningless because the next reply will re-expose it. Conversation-as-ACL is the original sin of most team-chat systems. We chose not to inherit it.

Section 05

The protocol wants to talk in deliveries. The CLI wanted to talk in deliveries too, at first. It worked. It was principled. It was also miserable.

Because a user — human or agent — does not think "I'm interacting with my delivery of this message." They think "I'm reading this message."

Section 06

The edge cases that almost broke it

The core model is clean. The edge cases are where design dies.

Active vs listed. An address can be routable without being visible in the directory. is_active controls routing. is_listed controls search. Conflating them was how we almost made "deactivate" also mean "erase from audit trails."

Inactive list vs empty list. Two distinct failures, two distinct error messages.

Reply-all uses logical headers. Not the frozen snapshot. If eng-leads@lists had three members at send time and five now, reply-all goes to the current five.

Self-send is allowed. You get both a delivery and a sent-item. They're independent.

Hide is a view filter, not ACL. Hidden messages still appear in direct read-by-ID. They just don't clutter your default list view.

Forward is a new message. Not a new delivery of the original. This protects no-retroactive-access.

Profile and directory collapse. Conceptually separate, physically merged in the MVP. Preserve the conceptual distinction, simplify the physical representation, document both.

BCC is deferred. The table exists. The invariants are locked. The CLI flag is not exposed until cryptographic hardening ships.

Section 07

Twelve verbs. No more.

Here is the entire MVP command surface:

That's it. That's the whole thing. Notice what's missing: search, forward, fork, snooze, archive, mark-unread, export, watch, mute, filter, escalate, move.

Agents will ask for all of these. We know they will. We're going to let them ask. Because we don't actually know which of those verbs matter most. So Inbox ships narrow, measures what agents try to do, and grows from evidence instead of intuition.

A design we rejected

We considered letting inbox ack default to "the last message you read." It felt natural. It was also a footgun. Hidden state that agents would have to model mentally. Behavior that would depend on session order — and would behave differently the moment you parallelized calls.

So we killed it. Every CLI command takes explicit IDs. No defaults that reach into session history. No "last message" magic.

Agents should never have to guess what state the CLI is carrying.

Section 08

How this design got hardened

The Inbox spec was built over dozens of conversations, and at every major inflection point, the current draft got handed to a different model — Claude, GPT-5.4, Gemini — with one instruction: try to break this.

The BCC privacy leak in the public header snapshot

Caught by Gemini. The original draft stored all recipients together, which would have exposed BCC'd addresses. We split into public/private recipient tables.

The parent-link metadata leak

Caught by Claude. If you can see a reply but not its parent, a naive implementation still returns parent_message_id, leaking the existence of a message you shouldn't know about. Fix: if parent isn't visible, omit the field.

The monotonic state machine mistake

Caught by Claude. We originally locked unread → read → acknowledged as one-way. Relaxed to mutable current state + append-only transition history.

The atomic send / partial fanout gap

Caught by Claude. "Send succeeds if deliveries created" wasn't precise enough. The answer: all or none, stated as an invariant.

The two kinds of review

Reviewers with full conversation history catch continuity issues — decisions made out loud but never written down. Best for design integrity.

Reviewers with fresh context catch implementer-facing ambiguities. If the docs don't survive cold reading, they don't survive handoff.

If your design is going to be implemented by agents, you should probably let agents help review it.

Section 09

Guess less. Instrument more.

We do not want to build features for agents. We want to build features agents have already told us they want. Not through interviews. Through their commands.

Every Inbox command emits structured telemetry. Just the shape of what's happening: which verb, which error code, how long it took, how many recipients resolved. That's the baseline. The interesting part is what happens when the agent types a verb that doesn't exist.

Section 10

The "coming soon" trick

When you enable experimental discovery mode, the CLI starts advertising verbs it doesn't actually implement.

$ inbox forward msg_01H2X... --to carol@vps-1
{
  "ok": false,
  "error": {
    "code": "coming_soon",
    "message": "forward is not yet implemented",
    "hint": "if you wanted this, run: inbox give-feedback"
  }
}

The commands don't mutate state. They log the attempt — the verb, the flags, the session context — and return coming_soon.

Two halves of the same loop

Probes catch what agents attempt. Passive signal.

Feedback catches what agents wished they could attempt, but couldn't guess the syntax. Active signal.

Together, they're the closest thing to reading agents' minds we're going to get.

Section 11

The docs are the diagnostic

Inbox has ten spec docs. For a local-first SQLite CLI with twelve commands, that might sound absurd. It's not. It's diagnostic.

The specs aren't documentation about the system. The specs are the system, and the code is an implementation of them.

Section 13

What this is really about

Agents aren't better when they have more tools. They're better when they have better primitives.

A tool is "call this API." A primitive is "this is how you talk to another agent, for real, durably, with everything we've learned about how humans talk to each other over thirty years of email." One is a capability. The other is a substrate.

Inbox is a bet that the substrate matters more.

The future version may span multiple hosts and stronger cryptographic guarantees. The first version only needs to prove that agents work better when they have inboxes.

We think they do.
