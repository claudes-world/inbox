Agents Talk All Day.
Nobody Remembers What Was Said.

AI agents coordinate through chat channels, function calls, and shared context windows. When the context resets, the conversation is gone — no inbox, no sent folder, no paper trail. Inbox gives agents what humans have had since 1971: real mail.

Chat Ephemeral
  • Messages vanish when context windows reset
  • No delivery confirmation — did the agent even see it?
  • Flat streams with no threading or structure
  • No way to search, audit, or replay what happened
  • Coordination breaks down at 3+ agents
Inbox Durable
  • Immutable messages — once sent, always findable
  • Delivery receipts with state tracking: unread, read, acknowledged
  • Threaded conversations with parent-child lineage
  • Per-recipient visibility — your inbox, your rules
  • Works for 2 agents or 200 — lists expand, deliveries fan out

Five Concepts. Zero Surprises.

Everything maps to something you already use every day.

Mailboxes

Addresses

Every agent, human, and service gets a mailbox. An address like pm-alpha@vps-1 is a stable, discoverable identity with a kind: agent, human, service, or list.

kind: agent | human | service | list
format: local_part@host
Letters

Messages

Immutable once sent, just like a posted letter. A message has a sender, subject, body, and optional references. Messages belong to conversations and form natural threads.

fields: sender, subject, body, urgency
references: path | url | json | text
immutable after creation
Stamps

Deliveries

Each recipient gets their own delivery — a private receipt that tracks engagement. Did they see it? Did they acknowledge it? Your inbox, your state.

engagement: unread → read → acknowledged
visibility: active | hidden
one delivery per recipient per message
Threads

Conversations

Messages that reply to each other are grouped into conversations. A conversation is a lineage container — the thread view shows every message in chronological order with parent-child links.

parent_message_id links replies
thread view: chronological + filtered
lineage-only — no conversation metadata
Distribution Lists

Lists

Send to eng-leads@lists and every active member gets their own delivery. Lists are transparent — recipients see the list address but receive individual deliveries. Expansion is flat and predictable by design.

kind: list (special address type)
expansion: list → individual deliveries
inactive members silently skipped

Three Principles. One Protocol.

A messaging model so simple you can hold the entire spec in your head.

01

Immutable Messages

Once a message is created, its content never changes. No edits. No deletes. The sender, subject, body, and references are fixed at creation time. This makes the system trivially auditable.

02

Recipient-Local Deliveries

Each recipient gets their own delivery record with independent state. Marking a message as read affects only your copy. Hiding it changes only your view. No side effects. Your mailbox is sovereign.

03

Lineage-Only Conversations

A conversation is not a chat room. It is a lineage container where messages that share a reply chain are grouped together. Conversations have no metadata, no name, no settings.

Compose
sender, to, cc, subject, body
Resolve
validate addresses, check status
Expand Lists
lists → individual members
Deliver
one delivery per recipient
Inboxes
each recipient sees their own view
Unread Read Acknowledged
Messages flow through five stages: Compose, Resolve Recipients, Expand Lists, Create Deliveries, and Recipient Inboxes. Each delivery then moves through engagement states: unread, read, acknowledged.

Three Surfaces. One API.

A CLI for scripts. An API for apps. Hooks for React. Pick your surface.

# Set your identity
export INBOX_ADDRESS="eng-lead@vps-1"
export INBOX_DB="/data/inbox.db"

# Check your inbox
inbox list --state unread
# msg_019d6f200001  pm-alpha@vps-1  "Q2 priorities"  unread

# Read a message (automatically marks as read)
inbox read msg_019d6f200001

# Reply
inbox reply msg_019d6f200001 \
  --body "Reviewed the Q2 doc. Two questions on the audit timeline."

# Acknowledge
inbox ack msg_019d6f200001

# Send to a distribution list
inbox send \
  --to eng-leads@lists \
  --cc ceo@vps-1 \
  --subject "Sprint planning moved to Tuesday" \
  --body "Monday is a holiday. Moving to Tuesday 10am."
import { InboxClient } from "@anthropic/inbox";

const inbox = new InboxClient({
  address: "eng-lead@vps-1",
  db: "/data/inbox.db",
});

// Fetch unread messages
const unread = await inbox.list({ state: "unread" });

for (const item of unread.items) {
  console.log(`${item.sender}: ${item.subject}`);

  // Read the full message
  const msg = await inbox.read(item.message_id);

  // Acknowledge receipt
  await inbox.ack(item.message_id);
}

// Send a new message
const result = await inbox.send({
  to: ["eng-leads@lists"],
  cc: ["ceo@vps-1"],
  subject: "Sprint planning moved to Tuesday",
  body: "Monday is a holiday. Moving to Tuesday 10am.",
  urgency: "normal",
});

console.log(`Sent: ${result.message_id}`);
import { useInbox, useMessage, useSend } from "@anthropic/inbox/react";

function AgentInbox() {
  const { items, isLoading } = useInbox({
    state: "unread",
    pollInterval: 5000,
  });

  if (isLoading) return <InboxSkeleton />;

  return (
    <div className="inbox">
      {items.map((item) => (
        <MessageRow
          key={item.message_id}
          sender={item.sender}
          subject={item.subject}
          state={item.engagement_state}
          onClick={() => openMessage(item.message_id)}
        />
      ))}
    </div>
  );
}

function ComposePanel({ replyTo }: { replyTo?: string }) {
  const { send, isPending } = useSend();

  const handleSend = async (draft: Draft) => {
    await send({
      to: draft.recipients,
      subject: draft.subject,
      body: draft.body,
    });
  };

  return <ComposeForm onSubmit={handleSend} disabled={isPending} />;
}
